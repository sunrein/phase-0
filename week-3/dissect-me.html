Week 1 blog

Here we are at the beginning of Dev Bootcamp week one, and I feel like I've learned a lot. The scope of the development world gets bigger and bigger every day. I never knew there was so much to learn, but I'm really enjoying the process so far. One of the key things I learned about this week is version control. This is a way that developers can communicate changes that they make to files between one another, because version control is an effective record keeping strategy. In essence, version control is a series of snapshots of changes made to a file, such as modifications made to a series of code. Rather than tracking every single change individually, version control keeps track of a series of changes made, either by one person, or group members collaborating on a project.

This is more helpful when looking at changes because you are not seeing the minutae of every single syntax correction or other minor change. Rather, you can see a set of changes that were made, why they were made, and how they change the source material. For instance, if I were to proofread this blog post after submitting it, I can commit a set of changes stating that I corrected spelling and syntax errors generally. Version control keeps track of the new version of this file, rather than explaining each change made individually. Of course, it is good to be specific when explaining changes made to a file, but version control doesn't keep track of every keystroke like, for instance, the track changes setting on Microsoft Word does.

One of the main benefits of version control is that multiple people can modify the same file simultaneously. Rather than overwriting each other's work, version control tracks and merges all of the changes. This makes it very easy to review changes as they come, understand why they were made, and reference every change as they are committed to the version control hub, whether it be GitHub or something else.

Git is a program that allows changes to be committed and tracked. The nice thing about git is that you do not have to commit changes until you are ready to, and you must tell git that you are adding and committing changes in a two step process. This ensures that you are confident about the changes you are committing, or in other words, submitting, to the original file. Rather than overwriting previous work, git and GitHub merge the changes to the master branch and require an explaination of the changes made in order to make the reasoning behind the changes clear to all parties, including the creator, down the line.

The way that git does this is through branches. Branches are sets of changes that will eventually lead back to the original, or master, branch. They are called branches because while they connect to the original file, they to not affect change directly upon it, and the branches must be merged for the changes to be reflected on the original file. Branches can be reviewed and eventually merged through GitHub, which is a webstie that allows for remote storing and tracking of files and changes in repositories on the site. Git is the software used to add, commit, and push on a local device.

GitHub is a good way to store files and track changes because it is a website, so files can be accessed remotely. This makes things like big open source projects possible, because people from all over the world are able to access the files and submit their own changes. On smaller scales, groups working remotely can access and contribute to the same files without having to share the files between one another directly. The files can be accessed simultaneously, and the changes can all be merged into the master branch. Therefore, using GitHub is ideal for working on projects among many people. However, there are benefits to working on solo projects using GitHub. The main benefit is version control; being able to review changes that were made and why, in case there are mistakes that need to be changed later. It is a great way of keeping a record of the evolution of a project from inception, and every step of changes along the way.
